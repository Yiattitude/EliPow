# Elipow V1 PRD (Product Requirements Document)

## Problem Statement

电力专业的本科生在学习过程中，面临着缺乏深度知识关联、无法将理论与真实岗位需求对齐的困境。现有的在线学习工具往往是“一刀切”的课程推送，缺乏个性化的查漏补缺机制；且现有的 AI 助手通常直接给出答案，剥夺了工科生独立推导和批判性思考的过程，且内容缺乏权威行业标准的溯源保障。

## Solution

Elipow 是一款面向电力本科生的智能学习助手。它通过建立包含行业标准的**全量知识图谱**，并基于目标岗位的**岗位路径**，为每一位用户维护专属的**个人动态图谱**。系统摒弃了手动编排计划的繁琐，利用 AI 基于用户的“时间预算”自动推断并生成每周**动态学习计划**。同时，系统内置一个**单体苏格拉底 Agent**，它在用户遇到难题时绝不直接给答案，而是通过提问一步步引导用户，只有在用户连续答错时才兜底给出解答，并且所有的回答都通过 GraphRAG 技术精准溯源到原版权威文档。

## User Stories

1. As a student, I want to 登录系统并选择我的“目标岗位”和“薄弱科目”，so that 系统能为我初始化精准的个人知识图谱权重。
2. As a student, I want to 输入我本周可用的“时间预算”，so that AI 可以自动在我的岗位路径上截取并生成恰好符合时间要求的最优学习任务。
3. As a student, I want to 在仪表盘上看到“主干+支线”的横向学习时间轴，so that 我能清晰地知道下一步该学什么，以及它们之间的依赖关系。
4. As a student, I want to 在完成某个知识点后，看到岗位技能树中对应的节点被高亮“充能”，so that 我能获得游戏化的正向反馈和成就感。
5. As a student, I want to 能够在一个全屏沉浸式的图谱工作台中浏览全量行业知识（支持缩小聚类和放大展开），so that 我能俯瞰整个专业的宏观架构。
6. As a student, I want to 一键切换“全量视角”与“个人掌握度视角”，so that 我能在图谱上直观看到哪些知识是暗淡的（薄弱），哪些是绿色的（已掌握）。
7. As a student, I want to 点击图谱上的任意节点，能在侧边栏查阅该节点的教学视频或规范文档，so that 我能立即开始学习。
8. As a student, I want to 唤出一个悬浮在图谱上的 AI 聊天框，并在点击不同节点时上下文能自动跟随，so that 我在问问题时不会丢失当前的图谱焦点。
9. As a student, I want to 得到 AI 的启发式提问而非直接答案，so that 我能自己推导出电路或系统的运行原理。
10. As a student, I want to 在我连续 3 次回答错误时，AI 能直接给我公式和解答，so that 我不至于卡在一个难点上完全无法推进。
11. As a student, I want to 点击 AI 回答中的引用标签 `<ref>`，右侧能自动跳转并高亮 PDF 原文，so that 我能核实知识的权威性。

## Implementation Decisions

- **分工架构**：项目被垂直切分为三大模块（用户成长引擎、知识图谱沉浸工作台、单体苏格拉底 Agent 问答系统），每个模块由开发者负责端到端（前端+后端+DB）实现。
- **图谱解耦与存储**：全量行业图谱的拓扑结构存储在 Neo4j 中；而个人的节点掌握度数据（0-100%权重）存储在 MySQL 的 `User_Node_Mastery` 表中。在查询“个人动态图谱”时，通过业务层将 Neo4j 的拓扑与 MySQL 的权重进行 Join，避免图数据库数据爆炸。
- **岗位图谱的定义**：明确“岗位能力模型/岗位路径”只是行业通用图谱的一个子集视图，并非独立数据结构。
- **学习计划生成**：舍弃用户纯手动编排，采用后端 `StudyPlanService` 基于“时间预算”和“掌握度”的动态推断算法。
- **AI 架构降级**：V1 严格遵循 MVP 原则，**放弃多智能体路由**，采用单一的基于 Spring AI 的 Socratic Agent，配合强大的 System Prompt 和 Function Calling 机制完成任务。

## Testing Decisions

- **测试重点 (Black-box)**：重点测试后端的外部 API 行为与前端的端到端渲染。
- **模块 1 测试**：测试 `StudyPlanService` 的推断逻辑，确保给定的时间预算（如10h）输出的任务序列耗时总和与之匹配，且前置依赖未掌握的节点不能被跳过。
- **模块 2 测试**：测试图谱组装逻辑，验证同一个 Neo4j 拓扑结构在传入不同 `user_id` 时，返回的节点高亮状态数组是否正确。
- **模块 3 测试**：测试 AI Prompt 的边界情况，模拟用户故意回答错误 3 次，断言 AI 是否准确切换态度给出最终答案。

## Out of Scope

- 教研反哺分析大屏（供教师端使用的群体画像和课程大纲对比工具）。
- 多智能体 (Multi-Agent) 协同引擎。
- 复杂的社交与排行榜系统。

## Further Notes

- 前端图谱的性能挑战较大，G6 的按需加载机制需要在后端接口层面予以配合（图局部展开）。
- Agent 的响应流式输出必须支持 LaTeX，因为涉及大量电力计算公式。
