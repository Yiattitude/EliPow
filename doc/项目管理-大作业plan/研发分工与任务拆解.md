# V1 研发分工与任务详细拆解表 (Vertical Slicing)

基于“垂直业务切分”、“单体Agent”、“基于图谱的AI动态路径”等核心架构共识，我们将系统拆分为三大全栈业务模块。
以下是每个模块**具体需要完成的代码开发与逻辑实现任务**，以便开发人员可以直接作为需求清单进行排期。

---

## 模块 1：用户成长引擎 (User Growth Engine)
**职责定位**：负责“门面”和“日常学习管理”。用户每天一睁眼看到的进度、打卡、学习路径规划都由这个模块负责。

### 1.1 前端具体要干什么？(React + Vite)
- **搭建基础脚手架**：引入 Shadcn UI 组件库，配置 Tailwind CSS 和基础路由。
- **冷启动画像采集卡片**：在用户初次登录后，用分步卡片（Onboarding）引导用户选择“年级”、“目标岗位（如：国网调度员）”以及“薄弱科目”。这些数据要一次性提交给后端。
- **高交互“主干+支线”时间轴 (Dashboard Timeline)**：
  - **不要用**传统的垂直列表展示进度。
  - 必须使用 `@xyflow/react` (React Flow) 渲染一条横向的路径图。
  - 用户能在路径图上看到本周要学习的 5-10 个节点（分主线和支线）。
  - 提供“拖拽”功能，允许用户微调节点的先后顺序。
- **岗位技能树 (Skill Tree)**：
  - 在仪表盘左下方做一个面板，展示用户当前的技能充能进度（类似游戏天赋树）。
  - 引入 Lottie 或 Spine 2D 动画库，当某个节点完成时，播放“充能点亮”的特效。
- **每周预算弹窗**：每周初强制弹窗，让用户输入“本周计划投入几小时”，并将该时间发给后端。

### 1.2 后端具体要干什么？(Spring Boot + MySQL)
- **账号与鉴权系统**：集成 Spring Security 和 JWT。建立标准的 `User` 表。预留 OAuth2 接口用于后续对接微信/QQ扫码。
- **用户画像表定义**：设计 `User_Profile` 表，存储用户选定的目标岗位和薄弱项。
- **核心算法：动态周计划推断引擎 (StudyPlanService)**：
  - **输入**：用户本周的“时间预算”（如 10 小时），以及用户的目标岗位。
  - **逻辑**：读取用户当前的“图谱掌握度”数据，找出该目标岗位路径上**权重最低且具备前置条件**的节点。根据每个节点预估的耗时，截取大约 10 小时工作量的节点集合。
  - **输出**：生成本周的任务序列 `WeekPlan` 并持久化到数据库，返回给前端 Timeline 渲染。

## 模块 2：知识图谱沉浸工作台 (Knowledge Graph Explorer)
**职责定位**：整个系统的数据基石和“大地图”。负责底层知识结构的管理和全景可视化呈现。

### 2.1 前端具体要干什么？(React + AntV G6)
- **图谱核心渲染器**：
  - 引入蚂蚁金服的 `AntV G6` 库搭建 2D 拓扑网络。
  - 必须实现**按需加载 (Lazy Loading)** 和 **语义缩放 (Semantic Zooming)**：因为全量图谱可能有上千个节点，缩小时只能看到“电路原理”、“电机学”等大圆球，放大后才展示具体的细分知识点。
- **全局/个人视角的无缝切换**：
  - 顶部的 Toggle 开关。默认是“全量图谱视角”（展示行业所有知识）。
  - 当切换到“个人视角”时，调用平滑动效，将用户尚未掌握的节点变暗/半透明，将已掌握的节点点亮成绿色。
- **侧边透视面板 (Sidebar)**：
  - 当用户在图谱中点击某个具体节点时，右侧划出抽屉 (Drawer)。
  - 抽屉内展示该节点对应的学习资源（视频链接、PDF 文档）、所属岗位标准等详情。
- **What-If 预演交互提示**：如果用户强行跳过某个前置节点，前端需弹出红色警告：“跳过此节点将导致后续[某某节点]理解率下降30%”。

### 2.2 后端具体要干什么？(Spring Boot + Neo4j)
- **图谱数据库搭建与建模**：
  - 部署和配置 Neo4j 图数据库。
  - 使用 Spring Data Neo4j 定义节点实体 (NodeEntity)：区分五级（领域、标准、知识点、场景、步骤），并定义它们之间的依赖关系 (Relationship)。
- **图谱数据查询 API**：
  - 开发能按层级、按领域拉取子图的接口，防止一次性拉取全量数据导致前端崩溃。
- **“个人图谱同步”的数据结构设计 (关键)**：
  - **千万不要**为每个用户在 Neo4j 里复制一份全量图谱（数据会爆炸）。
  - **正确做法**：在关系型数据库 (MySQL) 中建一张映射表 `User_Node_Mastery (user_id, node_id, mastery_level, last_updated)`。当查个人图谱时，从 Neo4j 查拓扑结构，从 MySQL 查掌握度，在内存中拼装（或直接交由前端关联）后返回。
- **岗位路径生成 API**：写图查询语句 (Cypher)，输入“电气工程师”，利用图算法遍历并抽取出一条从基础到高级的必经路径子集。

## 模块 3：单体苏格拉底 Agent 问答系统 (Single-Agent Socratic QA)
**职责定位**：核心的“虚拟助教”。它不仅要回答问题，更要通过提问来引导用户，并且必须能给出准确的资料来源。

### 3.1 前端具体要干什么？
- **沉浸式聊天面板**：
  - 开发一个聊天窗口。支持两种模式：独立全屏模式、图谱页面的悬浮/画中画 (PiP) 模式。
  - 面板顶部需要有一个“跟随图谱焦点”的开关，打开时，聊天室的上下文自动锁定在当前图谱选中的节点上。
- **流式输出与 Markdown 渲染**：
  - 接入后端的 SSE 流，像 ChatGPT 一样逐字打字机效果输出。
  - 支持渲染复杂的 Markdown 和 LaTeX 数学公式。
- **精准溯源高亮 (PDF 联动)**：
  - 解析大模型返回的特定标签（例如 `<ref id="doc_page_3">`）。
  - 在聊天窗右侧集成一个 PDF/文档阅读器，当用户点击引用标签时，右侧阅读器自动滚动到那一页，并将该段落高亮显示。
- **一键转为闪卡**：在每轮优质对话旁边加一个按钮，点击后将核心内容截取为卡片，存入用户的个人复习库中。

### 3.2 后端具体要干什么？(Spring AI + 向量数据库)
- **向量数据库与知识分块 (Chunking)**：
  - 引入 Milvus 或 PGVector 向量数据库。
  - 写一段离线脚本：将现有的教材 PDF、行业标准文档进行切片 (Chunking)、向量化 (Embedding) 后存入数据库，并与图谱的节点 ID 建立映射关系。
- **单体 Agent 编排 (Spring AI)**：
  - 用 Spring AI 搭建大模型调用的底层代码。
- **核心大工程：调优 System Prompt**：
  - 必须写出极其严密的系统提示词，强制要求模型：**绝对不能直接给出最终答案，必须以苏格拉底式提问来引导用户推导。**
  - **挫折感知机制**：在提示词或上下文中记录错误次数，当用户对同一知识点连续 3 次回答错误时，触发降级策略，模型改用“直接讲解+给出公式”的模式。
- **Function Calling (工具函数调用)**：
  - 给模型配备一组可调用的函数，例如：
    - `searchVectorDB(query)`：去向量库搜文档，以便大模型拿到原文引用并生成 `<ref>` 标签。
    - `getUserMastery(userId, nodeId)`：查询这个用户对当前节点的掌握度。
    - `updateUserMastery(userId, nodeId, delta)`：当大模型通过问答确认用户已掌握时，调用此函数给用户的图谱权重加分。
