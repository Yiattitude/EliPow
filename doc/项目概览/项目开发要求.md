# AGENTS.md

## 1. 项目名称
**电荔——上电AI学习助手**

## 2. 项目定位
本项目是一个面向**电气相关专业本科生**的**个性化学习成长系统**。  
系统通过用户画像、知识图谱、能力测评、学习路径推荐、行业资讯联动推送和基于资源复用的智能问答，帮助学生明确“当前处于什么学习阶段”以及“下一步应该学什么”。

## 3. 项目目标
- 帮助电气相关专业本科生建立清晰的专业学习路线
- 根据用户当前能力水平，生成个性化学习计划
- 将课程资料、知识库、行业资讯和学习资源统一组织起来
- 基于已有资源实现专业问答，而不是单纯依赖大模型自由生成
- 实现行业资讯与学习内容的联动推荐
- 为学生提供持续成长的学习闭环：**测评 → 推荐 → 学习 → 反馈 → 再推荐**

## 4. 核心用户
- 电气相关专业本科生
- 主要面向上海电力大学学生场景
- 重点关注专业学习、课程补强、技能提升、竞赛准备、行业认知

## 5. 项目范围边界
### 5.1 做的内容
- 用户注册、登录、画像采集
- 专业能力测评
- 学习路径推荐
- 学习资源推荐
- 专业知识问答
- 行业资讯抓取与联动推送
- 学习进度记录
- 知识图谱展示

### 5.2 不做或后期再做的内容
- 社交/人脉查找
- 复杂社区互动
- 课程交易/付费内容
- 多角色复杂审批系统
- 过度复杂的多智能体自治系统
- 泛人群教育平台
- 面向初中生的科普体系

## 6. 核心产品理念
### 6.1 问答不是主线
问答功能的定位是：**对已有学习资源和知识库的再利用**。  
系统不是只做“聊天机器人”，而是要服务于“学习成长路径”。

### 6.2 个性化是主线
系统的核心是：
- 识别用户当前水平
- 找出知识缺口
- 推荐下一步学习内容
- 持续更新学习计划

### 6.3 行业资讯要和学习内容联动
行业新闻不能只是信息流展示，而要能和用户当前学习的知识点建立关联，例如：
- 用户正在学习新能源并网
- 系统推送相关政策、案例、企业动态、技术进展

## 7. 技术栈
### 7.1 前端
- React
- Tailwind CSS
- shadcn/ui
- Lucide Icons

### 7.2 后端
- Spring Boot
- Spring Security
- MySQL
- Redis

### 7.3 AI 智能体
本项目的 AI 智能体能力使用 **Spring AI** 实现。  
AI 侧优先采用 Spring AI 提供的：
- `ChatClient`
- `Advisor`
- `PromptTemplate`
- Tool Calling / Function Calling
- RAG（检索增强生成）
- Chat Memory / Conversation History

### 7.4 推荐架构
- **Java 后端**：负责业务系统、用户系统、数据存储、权限、资源管理
- **Spring AI 模块**：负责问答、路径生成、资源推荐、资讯联动、工具调用
- **前端**：负责交互、可视化、学习路径展示、图谱展示

## 8. AI 架构建议
### 8.1 推荐方案
采用 **“主智能体 + 工具调用 + RAG + 规则引擎”** 的轻量架构，而不是一开始就做复杂多智能体系统。

### 8.2 主智能体职责
- 判断用户意图
- 调度相应工具
- 组织最终回答
- 保证输出结构化、可解释

### 8.3 工具能力
建议具备以下工具：
- 用户画像查询
- 知识库检索
- 学习路径生成
- 学习资源推荐
- 资讯抓取与分类
- 测评结果查询

### 8.4 RAG 作用
RAG 用于：
- 专业问答
- 资源再利用
- 从课程资料、笔记、文档、新闻中提取答案依据

## 9. Spring AI 使用约定
### 9.1 使用方式
- 通过 `ChatClient` 统一处理对话请求
- 通过 `Advisor` 注入系统提示、检索上下文、历史对话
- 通过 Tool Calling 让模型调用业务接口
- 通过 RAG 检索知识库后再生成答案

### 9.2 约束原则
- AI 输出必须尽量结构化
- AI 回答必须优先依赖检索内容和业务数据
- 不允许完全无依据的自由发挥
- 复杂业务规则交给后端和规则引擎控制

### 9.3 输出要求
AI 输出尽量返回统一结构，例如：
- `current_level`
- `gap_topics`
- `next_steps`
- `recommended_resources`
- `explanation`
- `sources`

## 10. 功能模块
### 10.1 用户画像模块
采集用户信息：
- 年级
- 专业方向
- 学习目标
- 当前课程状态
- 自评水平
- 测评结果
- 学习行为记录

输出：
- 当前能力层级
- 学习偏好
- 推荐方向
- 缺失知识点

### 10.2 专业能力测评模块
通过题目或任务判断用户当前所在水平。  
测评结果用于驱动学习路径推荐。

### 10.3 知识图谱模块
构建电气相关专业知识图谱，例如：
- 电路基础
- 模拟电子技术
- 数字电子技术
- 电机学
- 电力系统分析
- 继电保护
- 电力电子技术
- 新能源发电
- 智能电网

### 10.4 学习路径推荐模块
根据：
- 用户画像
- 测评结果
- 学习目标
- 知识图谱依赖关系

生成：
- 当前阶段建议
- 下一步学习内容
- 每周学习计划
- 进阶路线

### 10.5 学习资源推荐模块
推荐资源类型：
- 课程视频
- 文档资料
- 习题
- 书籍章节
- 案例分析
- 行业文章

### 10.6 专业问答模块
用户提问后：
1. 先检索资源库
2. 再由大模型生成回答
3. 回答中尽量给出引用来源或依据

### 10.7 行业资讯联动模块
从行业新闻、政策、企业动态中抓取内容，并与知识点标签匹配，形成学习关联推送。

### 10.8 学习进度模块
记录用户：
- 已学知识点
- 完成任务
- 测评变化
- 推荐点击与学习情况

优先实现：
1. 用户登录与画像
2. 知识图谱基础结构
3. 学习路径推荐
4. 专业问答
5. 学习资源推荐

### 第二阶段：增强
6. 行业资讯联动推送
7. 学习进度与反馈
8. 图谱可视化

### 第三阶段：优化
9. 更精细的个性化推荐
10. 学习行为驱动的动态路径调整
11. 更丰富的资源来源整合

## 12. 数据来源与内容原则
### 12.1 内容来源
- 课程资料
- 专业知识库
- 学习文档
- 行业资讯
- 标准/案例/公开资料

### 12.2 原则
- 内容要可追溯
- 问答结果要尽量基于检索内容
- 不允许完全无依据的胡编
- 推荐结果要可解释

## 13. 交互设计原则
- 页面风格简洁、现代、专业
- 优先突出“学习路径”和“当前状态”
- 首页应该展示：
  - 当前等级
  - 推荐下一步
  - 最近资讯
  - 学习进度
- 图谱展示作为亮点模块，但不要喧宾夺主

## 15. 后端实现建议
### Spring Boot 模块划分
- auth：登录认证
- user：用户画像
- assessment：能力测评
- knowledge：知识图谱
- learning-path：学习路径
- resource：资源管理
- qa：智能问答接口
- news：资讯抓取与联动
- progress：学习进度
- admin：后台管理
- ai：Spring AI 智能体接口层

## 16. AI 智能体实施建议
### 16.1 推荐智能体模式
- 单主智能体
- 多工具调用
- 结合 RAG
- 结合规则引擎
- 统一输出结构化结果

### 16.2 核心能力
- 问答
- 学习路径生成
- 资源推荐
- 资讯联动
- 解释型回答

### 16.3 不建议
- 一开始做复杂多智能体互相协作
- 让模型完全自由决定用户画像
- 让模型完全自由生成学习规划而不受规则约束

## 17. 非功能要求
- 回答尽量准确、可解释
- 页面风格统一
- 接口职责清晰
- AI 与业务逻辑分离
- 支持后续扩展
- 不追求复杂但不稳定的多智能体体系

## 18. 风险提示
- 不要把“AI”当成万能词，产品核心仍然是知识体系和学习路径设计
- 不要把问答做成唯一重点
- 不要让系统面向过宽人群
- 不要过度依赖大模型自由生成，必须配合规则和检索
- Spring AI 适合工程集成，但要避免把所有逻辑都塞进 AI 层

## 19. 项目成功标准
一个合格的版本应该能够：
- 正确识别用户大致学习水平
- 给出合理的下一步学习建议
- 能基于资源库回答专业问题
- 能把资讯和学习内容关联起来
- 有清晰的学习成长闭环
- 界面像一个真正可用的产品，而不是实验性 demo

## 20. 开发协作说明
开发时优先保证：
1. 主流程可用
2. 数据结构清晰
3. 输出可解释
4. 页面体验完整
5. AI 能力可控

如果遇到功能冲突，优先保留：
- 学习路径推荐
- 用户画像
- 专业问答
- 资源推荐

其余功能作为增强项处理。
