# 数据库设计文档

## 1. 设计原则
- 采用 **关系型(MySQL) + 图数据库(Neo4j)** 混合存储架构。
- MySQL 负责扁平化的核心业务数据存储（如账号、订单级记录等）。
- Neo4j 负责复杂关系网络存储，支撑 GraphRAG 极速穿透与多智能体推理。

## 2. 存储选型划分

### 2.1 MySQL (关系型存储)
- **用户表**：基础账号信息、权限配置。
- **配置字典表**：系统基础配置项。
- **行为流水表**：用于大批量、结构统一的学习日志和事件埋点持久化。

### 2.2 Neo4j (图数据库存储)
- **行业通用知识网络（五级分层）节点**：
  - [领域 Node] (如：智能电网)
  - [岗位 Node] (如：巡检工程师)
  - [知识点 Node]
  - [场景 Node] (真实企业案例映射)
  - [步骤 Node]
- **通用关系 (Edges)**：包含(HAS)、依赖(DEPENDS_ON)、前置(REQUIRES)、解决(SOLVES)等。
- **个人动态图谱**：
  - [用户 Node]
  - 用户的 [掌握关联 Edge] (例如 `(User)-[MASTERS {weight: 0.8}]->(KnowledgePoint)`)
  - 用户的 [学习进度关联 Edge]

## 3. 关系概述 (以图为主)
- 岗位 -> 要求 -> 知识点集合
- 知识点 -> 构成 -> 场景/步骤
- 用户 -> (伴随时间轴演化的权重) -> 知识点
- 大纲体系 -> 映射对比 -> 岗位需求图谱 (用于教研反哺)

## 4. 约束
- 知识图谱节点的增删改查优先在 Neo4j 中保证事务性。
- GraphRAG 检索出的子图，必须提供明确的节点唯一标识，供多智能体回溯上下文。
